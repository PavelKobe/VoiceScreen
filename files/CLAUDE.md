# CLAUDE.md — VoiceScreen MVP

> Этот файл читается Claude Code при работе с репозиторием. Сохраняй его в актуальном виде — он определяет контекст для всех подкаталогов.

## Что это

**VoiceScreen** — голосовой AI-агент для первичного скрининга кандидатов на массовых позициях (курьеры, кассиры, комплектовщики). Клиент (HR) загружает список откликов — агент обзванивает, задаёт анкету, возвращает шорт-лист.

## Стадия проекта

MVP, part-time разработка. **Оптимизация — на скорость MVP, а не на архитектурную чистоту.** Упрощения приветствуются, преждевременные абстракции — нет.

## Технологический стек

- **Язык:** Python 3.11
- **Веб-фреймворк:** FastAPI + Uvicorn
- **БД:** PostgreSQL 15 (через SQLAlchemy 2.x async)
- **Очереди:** Redis + Celery
- **STT:** Yandex SpeechKit (streaming)
- **LLM:** OpenAI GPT-4o-mini (основной), YandexGPT (fallback для РФ-контура)
- **TTS:** Yandex SpeechKit (голоса `alena`, `ermil`)
- **Телефония:** Mango Office (webhook + API)
- **Хранилище записей:** Yandex Object Storage (S3-совместимое)
- **Клиентский интерфейс на MVP:** Telegram-бот (веб-кабинет — позже)
- **Хостинг:** Yandex Cloud VM, Ubuntu 22.04, Docker Compose

## Конвенции кода

- **Async везде**, где делаем I/O. SQLAlchemy 2.x async, httpx async, FastAPI async.
- **Типизация строгая.** Все публичные функции с аннотациями. `mypy --strict` в CI, когда дойдёт.
- **Форматирование:** `ruff` + `black` (конфиги в `pyproject.toml`).
- **Логирование:** `structlog` в JSON-формате. Не использовать `print()`.
- **Секреты:** только через переменные окружения и `pydantic-settings`. Никогда не коммитить `.env`.
- **Ошибки внешних API:** всегда с retry через `tenacity` (3 попытки, экспоненциальный backoff).
- **Миграции БД:** только через Alembic, руками в БД ничего не менять.

## Что НЕ делать (сознательные упрощения MVP)

- Не добавлять Kubernetes, ArgoCD, service mesh. Docker Compose достаточно.
- Не писать свой STT/TTS. Только API.
- Не делать микросервисы. Один монолит, один Dockerfile.
- Не добавлять GraphQL. REST + webhooks.
- Не добавлять фронтенд. На MVP только Telegram-бот для клиентов.
- Не абстрагировать LLM/STT/TTS в интерфейсы «на будущее», если нет второй реализации.
- Не писать сложные юнит-тесты для CRUD. Тесты — только для диалоговой логики и платной обработки.

## Критичные вещи для MVP

1. **Сквозной тестовый звонок** (`make test-call`) должен работать на любой машине после `make install`.
2. **Первая фраза агента** обязана содержать уведомление о записи и возможность отказа (152-ФЗ, см. `docs/`).
3. **Логирование транскриптов** — в БД сразу по ходу диалога, не в конце. Если процесс упадёт, данные не должны теряться.
4. **Rate-limit на исходящие звонки** — не более 3 попыток на кандидата, не раньше 9:00 и не позже 21:00 локально.

## Текущие приоритеты

См. `docs/ROADMAP.md`. На данный момент фокус — **сквозной звонок с YAML-анкетой и записью в БД**. Всё остальное — после.

## Рабочий процесс с Claude Code

- При работе в подкаталоге сначала читай локальный `CLAUDE.md`, если он есть — там специфика.
- При добавлении новых интеграций (новая телефония, новый LLM-провайдер) — сначала обсуждение в PR, не в коде.
- Промпты для LLM живут в `app/core/prompts/`, не в коде. Это критично для итераций.
- Перед любым изменением в `app/core/dialog.py` прогнать `make test-call` до и после.
